# Client delivery

`ippan-pv-agent` is designed to be delivered to energy clients as a
self-contained component. A typical pilot delivery contains:

1. the `pv-agent` binary (or this source repository under NDA);
2. an `examples/pv/pv-agent.example.toml` tuned for the client's plant;
3. a written runbook (this document);
4. a key-management note describing how the operator key was provisioned.

## Install

### From source

```bash
cargo build --release
sudo cp target/release/pv-agent /usr/local/bin/pv-agent
```

### From a delivered binary

Copy `pv-agent` to a location on `PATH` (`/usr/local/bin/`,
`C:\Program Files\IPPAN\pv-agent\`, etc.) and make it executable.

## Configure the plant

```bash
pv-agent init --out /etc/ippan/pv-agent.toml --plant-id <YOUR_PLANT_ID>
```

Edit `/etc/ippan/pv-agent.toml`:

- `storage.base_dir` — durable storage for evidence bundles.
- `ippan.endpoint` — IPPANCENT L1 endpoint.
- `ippan.admin_token_env` — name of the env var holding the bearer token.
- `ippan.submit_anchors` — set `true` only when ready to anchor live.
- `key.key_file` — path to the operator key file (delivered separately).
- `agent.production_mode = true` — refuses demo keys.

## Provision a key

For a real pilot, the operator key should be generated by the operator's
own tooling and stored at the path referenced by `key.key_file`. For an
early demo, a local demo key is acceptable:

```bash
pv-agent generate-demo-key --out /etc/ippan/keys/demo-key.json
```

Treat the file as sensitive (`chmod 600` on Unix; restrict ACLs on
Windows). Demo keys are clearly marked `is_demo = true` and refused in
production mode.

## Smoke test

```bash
pv-agent demo --plant palermo-1mw
pv-agent verify --bundle data/pv-agent/palermo-pv-001/records/2026/05/15/pv-palermo-pv-001-20260515T101500Z
```

Both must succeed before going further.

## Run one real reading

```bash
pv-agent run-once \
  --input /var/lib/pv-agent/inputs/latest.json \
  --events /var/lib/pv-agent/events/recent.json \
  --config /etc/ippan/pv-agent.toml
pv-agent verify --bundle <printed-bundle-path>
```

## Anchor (when authorized)

```bash
export IPPAN_ADMIN_TOKEN=...
pv-agent anchor-submit \
  --bundle <bundle-path> \
  --config /etc/ippan/pv-agent.toml \
  --submit-anchor
```

`--submit-anchor` is **required** unless `ippan.submit_anchors=true` is
already in the config. Failing to set the env var is a hard refusal.

## Scheduling

A 15-minute scheduler can be run as a foreground process or, on Linux,
as a systemd timer. Example unit files are included in the same
directory as this document:

- [pv-agent.service.example](pv-agent.service.example)
- [pv-agent.timer.example](pv-agent.timer.example)

These are **examples**. They are not enabled automatically.

## What clients receive vs. what they don't

| Delivered                                       | Not delivered                          |
|-------------------------------------------------|----------------------------------------|
| `pv-agent` binary (or source under NDA)         | IPPAN AgentOS source                   |
| Canonical evidence schema documentation         | IPPANCENT L1 source                    |
| `examples/pv/*`                                 | Production endpoint admin tokens       |
| This runbook                                    | Other clients' keys or evidence        |
| The independent verifier — `pv-agent verify`    | Internal IPPAN ops dashboards          |

The agent may be proprietary, but the evidence must be independently
verifiable — and the verifier ships with the agent.
